# Equity Calculation

## Overview

Equity is calculated via Monte Carlo simulation: for a fixed hero hand and villain range,
thousands of random runouts are sampled and the win rate is measured.

The result is a number between 0 and 1 (e.g. 0.327 = 32.7% equity).
Ties (split pots) count as 0.5 wins.


## Algorithm (calculateEquity)

Input:
  - heroCards           — the hero's two hole cards
  - villainRangePercent — e.g. 40 for "Top 40%"; use 100 for a fully random villain
  - boardCards          — 0–4 community cards already dealt (empty for preflop)
  - simulations         — number of Monte Carlo iterations (default 1 000, scripts use 100 000–200 000)

Steps per call:
  1. Resolve the range: convert villainRangePercent to a list of hand types
     (e.g. ["AA","KK","AKs","AKo",...]) via getRangeForPercent().
     At 100% this returns all 169 hand types = fully random villain.
  2. Build valid villain combos: enumerate every suit-specific combo for each hand
     type in the range, then remove any combo that conflicts with heroCards or boardCards.
     Example: AKo produces 12 combos (4 suits for A × 3 remaining suits for K).
     This list is computed once and reused across all simulations.
  3. Per simulation:
     a. Pick a villain hand: draw uniformly at random from the valid combo list.
     b. Build remaining deck: all 52 cards minus hero, board, and villain cards.
     c. Shuffle remaining deck and deal the missing community cards (5 − |boardCards|).
     d. Evaluate both hands using bestHandScore() (best 5-card hand from 7 cards).
     e. Award +1 for a hero win, +0.5 for a tie, +0 for a loss.
  4. Return total wins / simulations.

The uniform draw over valid combos naturally weights each hand type by its combo count
(pairs 6, suited 4, offsuit 12), so no extra frequency correction is needed.


## Hand Evaluator (bestHandScore)

bestHandScore() takes any number of cards (≥ 5), finds the best 5-card hand, and returns
an integer score. Higher = stronger.

For exactly 7 cards it uses a tight double-loop (omit each pair of indices) to enumerate
all C(7,5) = 21 combinations without allocating extra arrays.

For exactly 5 cards it calls evaluate5() directly (no enumeration needed).

For other lengths it uses a recursive backtracking picker (rarely needed).

evaluate5() encodes the result as:

  score = category × 15^5 + rank_1 × 15^4 + rank_2 × 15^3 + ...

Categories (0–8): high card, one pair, two pair, three of a kind,
straight, flush, full house, four of a kind, straight flush.

Higher integer = stronger hand, so scores can be compared directly with >.


## Range Definition

All 169 starting hand types are ranked in HAND_RANKINGS (lib/range.ts),
ordered by strength from strongest (AA) to weakest (32o).

getRangeForPercent(p) walks the list and accumulates combos until the target
count (round(p/100 × 1326)) is reached. Combo counts: pairs = 6, suited = 4, offsuit = 12.
Total combos: 1326.

### Was wir selbst berechnen

1. Hand-Evaluator (lib/evaluator.ts): eigener Code, berechnet den Score einer
   5-Karten-Hand (Pair, Flush, Straight, etc.) via Integer-Encoding.

2. Monte-Carlo-Simulation (lib/equity.ts): eigener Code, mischt Deck, zieht
   zufällige Villain-Hand aus einer Range, vergleicht 7-Karten-Hände mit pokersolver.
   Liefert z.B. "AKs hat 67% Equity gegen top-40%".

3. getRangeForPercent() (lib/range.ts): eigener Code, liest HAND_RANKINGS von
   oben und schiebt Hände in die Range bis die Combo-Zahl das Ziel erreicht.

### Was wir von Equilab übernommen haben

Nur die Reihenfolge in HAND_RANKINGS — also welche Hände zu "top 5%", "top 10%"
usw. gehören. Equilab hat diese Grenzen in 5%-Schritten (5%–95%) veröffentlicht;
wir haben HAND_RANKINGS daraus konstruiert:

  - Bandzugehörigkeit: kommt 1:1 von Equilab (z.B. T7s ∈ top-40%, K4o nicht)
  - Feinreihenfolge innerhalb eines Bandes: kommt von unserer eigenen HU-Equity-Simulation

Equilab ist keine Dependency — wir haben ihre Referenz-Ranges einmalig als
Kalibrierungs-Daten genutzt (see: scripts/rebuild-hand-rankings.ts).

Genauigkeit vs. Equilab (Anzahl Hände pro %-Stufe): max. Δ = −2 bei 55%,
sonst 0 oder −1. Kleinstmögliche Abweichung, da offsuit-Hände immer 12 Combos
auf einmal addieren.

### Zirkularität und Poker-Meta

Hand-Rankings sind theoretisch zirkulär:
  - "Gute Hände" = hohe Equity gegen die gegnerische Range
  - Die gegnerische Range = die Hände die "gut genug" sind
  → Fixed-Point-Problem

GTO-Solver lösen das exakt: sie iterieren bis zur Nash-Gleichgewichts-Range,
wo jede Hand genau so oft gespielt wird dass kein Spieler durch Abweichen gewinnt.
Equilab's Ranges approximieren diesen Fixpunkt — deswegen übernehmen wir sie als
Referenz statt selbst zu iterieren.

Wenn die Meta sich ändert (z.B. alle spielen looser):
  - Die Rankings würden sich leicht verschieben
  - Suited connectors (T9s, 87s) gewinnen relativ zu High-Card-Händen (K4o, Q7o)
    weil Nut-Potential gegen weite Ranges wertvoller wird
  - Der Effekt ist klein für die obere Hälfte (AA bleibt AA)
  - Die Unschärfe liegt im Mittelfeld (~30–60%), wo HU-Equity von Equilab abweicht


## Hand Equity Matrix

The HandEquityMatrix DB table stores pre-computed equity for each of the 169 hand types
against a completely random villain (villainRangePercent = 100).

This is used to power the range matrix UI where every cell shows the hand's equity
vs. a random opponent — useful for calibrating the HAND_RANKINGS order.

Each row:
  - handType   — e.g. "AA", "AKs", "72o"
  - equity     — equity vs. random villain (0–1)
  - iterations — 200 000 simulations per hand type
  - updatedAt  — last generated timestamp

One representative combo is used per hand type (e.g. A♥A♠ for AA, A♥K♥ for AKs,
A♥K♠ for AKo). Blocker effects are negligible at 200k iterations.


## Pre-generated Hand Database

Training hands are not computed live during a session. Instead, scripts
pre-generate hands with 100 000 simulations each and store them in the database.
The training UI serves hands from this pool.

Scripts must be re-run whenever equity.ts, evaluator.ts, or range.ts changes.
